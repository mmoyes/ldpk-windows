1.2Trademarks For better readability, Autodesk® Flame® software will be written as &flame; in this document. Similarly, Foundry® Nuke® will appear as &nuke;. Apple® macOS® is abbreviated as &osx;, Microsoft® Windows® is abbreviated as &windows;. Linux® is the registered trademark of Linus Torvalds in the U.S. and other countries. The holders of these trademarks are kindly requested to contact us if this is insufficient. 2The LDPK The Lens Distortion Plugin Kit (LDPK) is a collection of C++-sources and -headers, GLSL-sources, python scripts and csh-scripts which allow you to export lens distortion data from &tde4; to other software, implement your own lens distortion models, or just explore the underlying mathematics. The current version (by the time this document becomes valid) is 2.0. Roughly speaking, the LDPK contains the following.
  • Ready-to-use Plugins for &nuke; versions ≥ 10.0, with source code
  • Shaders and GUI-specification files for &flame; matchbox nodes
  • Headers for implementing your own &tde4; lens distortion plugins
  • Python bindings and wrapper for &tde4;'s lens distortion models
  • Documentation on the Standard Models of &tde4;
3Exporting lens distortion data 3.1Lens distortion nodes for &nuke; 3.1.1Installing ready-to-use nodes There is a compiled plugin for each combination of lens distortion model, operating system and &nuke; version. Officially, we support &nuke; starting at version 10.0. If you need plugins for earlier versions (which are still usable, but not updated any longer) please contact us. Usually this is not necessary, but in case you modify the source code according to some special requirements there are some helpful files. The procedure for compiling on &linux; and &osx; is nearly the same, so we will handle them both at once in the next section. 3.1.3.1Linux, OSX There are scripts for compiling &nuke; nodes for &linux; and &osx; at ldpk

└── script

    ├── make_nuke_nodes.user.linux.csh

    └── make_nuke_nodes.user.osx.csh

 In order to compile the nodes for your version of &nuke; you should do the following:
  • Find out which compiler will generate plugins compatible to your version of &nuke;
  • Edit the script mentioned before, check for the compiler options and enter the include and library paths to your installation of &nuke;
  • Start the script. Your plugins will be in directory $LDPK/lib/nuke/[os]/Nuke[X.Y]

In order to export lens distortion data to &flame;, &tde4; creates a batch, which contains four nodes, two for resizing in order to handle overscan data, one for distortion removal and one for reapplying distortion. The distortion nodes consist of a shader, a GUI-specification file (XML) and a preset file (also XML). Shader sources and GUI-specification files are located at
$3DE4/sys_data/flame/<flame version>
in your &tde4; installation. If we have to update the files involved (e.g. for bugfixing), the LDPK contains the latest versions of these files: ldpk

 We are preparing a tutorial which demonstrates the typical workflow within &flame;. Assume your batch is going to be named myexport in target directory somewhere. Then the export script creates the following file structure: somewhere

├── myexport_batch

│   ├── add_margin.resize_node

│   ├── fingerprint

│   ├── redistort.1.glsl

│   ├── redistort.matchbox_node

│   ├── remove_margin.resize_node

│   ├── undistort.matchbox_node

│   └── undistort.xml

└── myexport_batch.batch

 The file fingerprint is introduced by the export script in order to avoid unintended overwriting of non-batch directories during data export, please ignore this. The other files reflect the structure of the exported batch, containing a resize node for adding overscan margin, an undistort node, a redistort node and a resize node for removing the overscan margin. In order to import in &flame; pick one of the items in menu Load in &flame;'s Batch editor, like e.g. Load and append. 4Tools 4.1Creating UV-maps The LDPK contains a tool for creating plain UV-maps with overscan. It is available for all three platforms. As of &tde4r6;, the tool is part of the release. In order to use, please start it in a console and read the output. It will create an &openexr;-file with two &float;-valued channels containing a two-directional ramp image, which is compatible with e.g. the ST-Map Node in &nuke;. ldpk
